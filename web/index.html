The quality engine — why the loop is worth running

At multiple points a different model reviews the work for correctness, complexity, and simplification before it can move forward. The point is not redundancy: spec-level, plan-level, and diff-level blind spots are genuinely different, and a model that did not write the work catches what the author cannot see.

02

State & memory

the .doruk/ layer

An LLM session is stateless. .doruk/ is a portable, agent-readable record of where the work stands and what the next move is, small enough to read first, structured enough to trust. Each run orients on it before touching code and writes back so work compounds. Skills live at skills/state-memory/.

Short-term · working memory

Where does the work stand?

STATE.md (the landscape) plus each feature's handoff.md. Built around the "what's the next move?" test so it stays a board, not a log. Trimmed every session.

Long-term · durable memory

What did we learn, and why?

feature.md records plus MEMORY.md / memory/ notes. Decisions and their WHY, gotchas, cross-feature learnings, written for future readers.

The folder convention: three tiers, no overlap, reference don't duplicate:

.doruk/
├── STATE.md                     # the landscape: thin index + backlog (read first)
├── MEMORY.md                    # durable cross-feature learnings (the index)
├── memory/                      # longer notes, one file per topic
└── features/
    └── NN-slug/                 # one folder per feature, permanent
        ├── feature.md           # durable record: orientation, decisions + WHY, gotchas
        ├── handoff.md           # this feature's live, in-flight handoff
        ├── task-NN-slug/        # optional: a delegated task's spec
        └── archive/             # scratch that was only true during the work
03

Delegation & isolation

hand a beat to a different model

Any beat can be handed to a different model, OpenAI Codex or Google Gemini, without giving up control. The isolation primitive is the git worktree: the delegate works in its own sandboxed checkout, and the driving agent reviews the diff before anything reaches main. Skills live at skills/delegation/.

RoleWhat it doesWrite access
Consultant
read-only		 Critiques a plan, an architecture, or existing code. Returns problems, alternatives, and agreements, makes no changes. This is the role ship's cross-model reviews use. none

How it all composes

The workflow is the spine: discuss finds the shape, align converges on a design, ship drives it to a review-ready PR with quality reviewed by multiple models at every gate. The state & memory layer is what the workflow orients on before it starts and writes back to after it finishes, so each run compounds. Delegation & isolation is how any beat gets handed to a different model safely, in a worktree, with the driving agent reviewing the diff before anything reaches main.
